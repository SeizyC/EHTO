# Product i18n v1 — Design Spec

**Status:** Living document — updated as v1 is implemented.
**Date:** 2026-06-17
**Owner:** EHTO (Fantagram Inc.)

## Goal

Make an EHTO plaza fully come alive in the visitor's language. Every generative
surface a member-facing plaza produces — member identities, ambient dialogue,
the user-interest→room-evolution loop, and news fuel — is produced **natively**
in the plaza's language, not translated from Korean. Korean plazas keep working
exactly as today.

## Core principle: native generation, never translation

A plaza has exactly one language. For a non-Korean plaza there is **no Korean
intermediate step anywhere**. Members are generated in the plaza language at
birth; the ambient model composes directly in that language; interest keywords
are extracted in that language. Claude (Opus) is natively multilingual, so this
is the same generation quality Korean enjoys today — only the language target
changes. Edge-translation of Korean output was considered and rejected (feels
translated, doubles cost/latency, leaves Korean names).

## Decisions (locked with user)

1. **Plaza language is set by the owner explicitly**, defaulting to the owner's
   resolved account language, and is editable later in world settings.
2. **Members are generated in the plaza language** (name, persona, speech style,
   backstory) — no per-language template pools, no translation.
3. **Language precedence:** saved user selection > IP auto-detect > `ko`.
   Saved selection wins.
4. **News in v1:** Korean plazas keep Naver; non-Korean plazas get **Google News
   RSS** (free, no API key, `hl`/`gl` params) for general headlines, plus a
   theme query when the plaza has a bias.
5. **Mixed real users in one plaza** see the plaza in **the plaza's language**
   (room language wins; we do not per-viewer translate).
6. **The user-interest → room-evolution loop is the heart of v1** and must run
   natively in the plaza language.

## Scope

### In v1
- Language plumbing: `profiles.language`, `worlds.language`, resolution lib, UI.
- Native member identity generation in the plaza language.
- Native ambient daily conversation.
- Native user-interest catch → room-evolution loop
  (`implicit-pref` + `persona-drift` + `world-bias`).
- News fuel: Naver for Korean plazas (unchanged); Google News RSS for
  non-Korean plazas, theme-aware.

### Out of scope (separate later spec)
- Sophisticated news routing/classification (Naver theme vs incident tiering,
  category caps per language beyond the simple Google RSS path).
- Music & video **share caption** localization
  (`music-share.ts` / `youtube-share.ts` `personaCaption`, slot labels,
  YouTube `regionCode`/`relevanceLanguage`).
- Translating or migrating existing Korean plazas/members (they stay `ko`).

## Language type & resolution

Reuse the existing `Locale` (`"en" | "ko" | "ja"`) from
[src/lib/about-content.ts](../../../src/lib/about-content.ts) as the canonical
language type — do not introduce a second enum.

New module **`src/lib/language.ts`**:
- `LANGUAGE_NAMES: Record<Locale, string>` → `{ ko: "Korean", en: "English",
  ja: "Japanese" }` for prompt directives.
- `resolveUserLanguage({ saved, country })` → applies precedence
  (saved → `countryToLocale(country)` → `"ko"`).
- Re-export `Locale` for server-side consumers so prompt/news/persona code does
  not import from `about-content` (a client-leaning module).

`countryToLocale` already exists in `about-content.ts:17-23`
(`KR→ko`, `JP→ja`, else `en`).

## Data model & migration

One migration `supabase/migrations/20260617000001_i18n_language.sql`:
- `profiles.language text` — nullable; null means "not chosen", fall back to
  detection. Set when the user explicitly picks a language while logged in.
- `worlds.language text NOT NULL DEFAULT 'ko'` — the plaza's content language;
  single source of truth for all generation in that plaza.

No language column on `members` or `messages`: a member's identity text is
generated in the owning plaza's language and stored as-is in `members.name` /
`members.persona` / `members.backstory`; message text is whatever language the
ambient engine produced. The plaza language governs everything.

`user_signals.topic_keyword` keeps storing the extracted keyword **in the plaza
language** (a plaza is single-language, so no normalization layer is needed).

## Locale persistence (saved-selection-first)

Current state: locale is client-only (`localStorage "ehto:locale"` via
[src/lib/use-locale.ts](../../../src/lib/use-locale.ts)); nothing in the DB.

Changes:
- When a logged-in user picks a language (the existing `LangToggle` / a settings
  control), persist to `profiles.language` via a small `PATCH /api/me/language`
  (or fold into an existing profile endpoint). Anonymous users keep using
  `localStorage`.
- On load, the resolved language is `profiles.language` (logged-in) →
  `localStorage` → IP → `ko`. `use-locale.ts` gains an optional saved-value
  input so the same precedence is honored client-side.

## Plaza language (owner explicit)

- **Creation:** the character/plaza creation flow gains a language choice,
  pre-filled from the owner's resolved account language; written to
  `worlds.language`.
- **Editing:** add `language` to
  [src/app/api/world/settings/route.ts](../../../src/app/api/world/settings/route.ts)
  PATCH body and to `updateWorldSettings()` in
  [src/lib/world-store.ts](../../../src/lib/world-store.ts) (`166-188`).
- **Reading:** add `language` to the `WorldInfo` type
  (`world-store.ts:7-19`) and
  [src/app/api/world/info/route.ts](../../../src/app/api/world/info/route.ts)
  so the client and all server jobs can read it.

## Native member identity generation

Today members come from hardcoded Korean templates
([src/lib/member-templates.ts](../../../src/lib/member-templates.ts) `24-101`):
Korean names + Korean `speech_style`/`backstory_seed`, mixed KR/EN affinity tags.

Change:
- Keep **archetypes language-neutral**: an archetype is `{ affinitySlugs:
  string[], seedConcept: string }` using internal english slugs
  (`"indie","night-owl","music"`), reused across all languages.
- At member seed/activation (`ai-pool.ts`, `world-seed.ts` path that creates
  member rows), generate the **surface identity** — `name`, `speech_style`,
  `backstory` — in `world.language` via one LLM call, then persist to the
  `members` row. So an English plaza yields English-named residents with English
  voices; a Japanese plaza yields Japanese ones.
- Sprite selection is unchanged (already language-neutral).
- Existing members are untouched; only newly seeded/activated members in
  non-Korean plazas use the new path. Korean plazas may keep the existing
  templates (archetypes can wrap them for `ko`).

## Conversation i18n

Thread `language: Locale` from `world.language` through the ambient pipeline:
[src/lib/ambient-loop.ts](../../../src/lib/ambient-loop.ts) →
`generateAmbientLine()` → `buildSystemPrompt()` in
[src/lib/member-reply.ts](../../../src/lib/member-reply.ts) (`52-169`).

- Keep **one** prompt builder. Extract the Korean instruction frame
  (section labels, conversational-form guidance) into a `Record<Locale, …>`
  frame dictionary and add an absolute rule: *"Write only in
  {LANGUAGE_NAMES[language]}."* Persona content is already in the plaza language
  (members are generated natively), so the frame + directive are sufficient.
- Localize duration text `formatJoinedAgo()` (`member-reply.ts:33-49`,
  currently `오늘`/`어제`/`3일 전`) into a per-language formatter.
- Localize the peer-relation hint text injected in `ambient-loop.ts:~362`
  (`같이 어울린 적 N회` + topics).
- `music-share`/`youtube-share` caption generation is **out of scope** for v1;
  those personaCaption calls stay Korean and are only invoked by the share jobs
  (which we do not localize yet). Where a non-Korean plaza would trigger a
  Korean-caption share, gate it off for non-`ko` plazas in v1 so no Korean text
  leaks into a non-Korean plaza.

## User-interest → room-evolution loop (the heart)

The loop already exists; make each step language-native:

1. User chats in their plaza (plaza language).
2. **Extraction** — `implicit-pref.ts` runs a Haiku call to pull
   `topic_keyword`s. Add `language` to its prompt so extraction happens in the
   plaza language and keywords are stored in that language in `user_signals`.
3. **Drift** — `persona-drift.ts` (`103-104`) adds top topics to a chosen
   member's `persona.affinity`. Because affinity now mixes neutral slugs and
   user-language keywords, the drift application and any prompt surfacing must
   treat user-language keywords as free-text interest tags (they already feed
   into the native prompt, so they render in-language naturally).
4. **Bias** — when a topic accumulates, `world-bias.ts`
   (`WorldBias` `18-21`, `biasPromptLine` `65-73`) tilts the plaza. Localize
   `biasPromptLine()` output via the same frame dictionary so the bias line is
   injected in the plaza language.

Net effect: an English user who keeps talking indie music gets English residents
who increasingly talk indie music in English, and a plaza that tilts that way —
all native.

## News fuel

- **Korean plazas:** unchanged — Naver via
  [src/lib/news-fetch.ts](../../../src/lib/news-fetch.ts) (`getNewsHeadlines`
  `231-262`, `QUERIES` `61`, `MAJOR_OUTLETS` `41-52`).
- **Non-Korean plazas:** new **Google News RSS** source
  (`news.google.com/rss/search?q=…&hl=<lang>&gl=<region>&ceid=<region>:<lang>`),
  no API key. Parse the RSS XML for title + source + pubDate; reuse the existing
  freshness filter and 30-min cache shape, with the cache key extended by
  `language`.
- Source routing happens by `world.language`: `ko → Naver`,
  `en/ja → Google News RSS`. Bias-aware fetch: when the plaza has a bias, add
  the bias artist/theme as an extra RSS query (`world-bias.biasNewsQueries`
  gains a language-aware variant that returns Google-RSS-friendly queries for
  non-Korean plazas).
- Headlines are injected into the ambient prompt exactly where Korean headlines
  are today (`ambient-loop.ts:~25-26`), in the plaza language.

## UI

- **Account language:** the existing `LangToggle`
  ([src/components/LangToggle.tsx](../../../src/components/LangToggle.tsx)),
  when used by a logged-in user, also persists the choice to
  `profiles.language`.
- **Plaza language selector:** a control in the creation flow and in world
  settings, pre-filled from account language, writing `worlds.language`.

## Components & boundaries (summary)

| Unit | Responsibility |
|------|----------------|
| `src/lib/language.ts` (new) | Canonical `Locale`, `LANGUAGE_NAMES`, `resolveUserLanguage` precedence |
| migration | `profiles.language`, `worlds.language` |
| `world-store.ts` / world APIs | carry `language` in read + settings PATCH |
| `member-templates.ts` + seed path | language-neutral archetypes → native identity at creation |
| `member-reply.ts` | language-aware prompt frame + directive, localized duration |
| `ambient-loop.ts` | thread `language`; localized peer hints; route news by language |
| `implicit-pref.ts` / `persona-drift.ts` / `world-bias.ts` | native interest extraction → drift → bias |
| `news-fetch.ts` (+ Google RSS source) | source routing by language |
| `use-locale.ts` / `LangToggle` / settings | saved-selection precedence + persistence + plaza selector |

## Testing strategy

- **Unit:** `resolveUserLanguage` precedence table (saved/IP/default); Google
  News RSS parser (fixture XML → headlines); `formatJoinedAgo` per language.
- **Integration:** seed a non-`ko` plaza → assert generated member rows contain
  non-Korean `name`/`speech_style`; run one ambient tick → assert output is in
  the plaza language (heuristic: script/charset check); extraction on an English
  message → English `topic_keyword`.
- **Regression:** a `ko` plaza behaves identically (Naver news, Korean members,
  existing prompts) — guard the existing ambient regression script.
- **Manual:** create EN and JA plazas; confirm daily chatter, a news-seeded
  line, and that talking about a topic shifts member affinity over a few ticks.

## Risks / open points

- **Prompt-frame quality** per language is authoring work, not a model limit;
  budget a review pass on the EN/JA frames.
- **Google News RSS** has no official SLA; treat failures as "no news this tick"
  (the loop already tolerates empty news) and cache aggressively.
- **Affinity tag duality** (neutral slugs + user-language free-text) must be
  handled so slugs never render raw to users; they only drive prompt context,
  while user-language keywords render naturally.
- **Share-job gating** for non-`ko` plazas in v1 must be airtight so no Korean
  caption leaks before the share-localization phase.
